# SUMI Changelog

Built on [Papyrix](https://github.com/pliashkou/papyrix) by Pavel Liashkov (@bigbag), itself based on CrossPoint Reader by Dave Allie.

## [0.6.2] — May 2026

Stability + X3 + upstream-fix release. The 0.6.0 on-device EPUB indexer turned out to be too risky on heavy books — the parser would `std::bad_alloc` mid-extend, abort with `-fno-exceptions`, and on the next boot the partial cache would replay the same crash on every page turn (Issue #23, reported by MQuinteiroS). This release retires on-device indexing, recovers gracefully when an old aborted-extend cache is found, fixes a separate spurious "concurrency violation" log line that was muddying crash diagnostics, and pulls three high-value patches from CrossPoint Reader 1.3.0 — including the JPEGDEC wild-pointer crash and the X3 grayscale-washout fix that lets X3 users finally turn AA back on.

**Heads-up on BLE page-turners.** Reliable BLE page-turning needs the book's pages to be laid out *before* read time so the parser doesn't run while NimBLE's ~48 KB sits resident in heap. With the on-device indexer gone, the only way to get there is to pre-render books via [sumi.page/process](https://sumi.page/process/) — drop the EPUB, get back an XTC bundle that the reader skips the parser for entirely. Books opened straight from `/convert/` will still work for tap-the-button reading; the prerequisite only matters once a BLE turner is attached. The converter page now surfaces this directly.

**Partial-cache crash recovery (Issue #23 fix)** — `PageCache::extend` writes a `<cachePath>.attempt` marker file to SD *before* invoking the parser's cold-extend path, and deletes it after a survived return. If the parser aborts mid-call (uncaught `std::bad_alloc` → `terminate` → `abort` under `-fno-exceptions`), the marker survives. On the next boot, `PageCache::load` sees the marker and seals the existing partial cache as complete — both in memory and on disk via an in-place rewrite of the `isPartial` byte at header offset 22 (non-truncating `O_RDWR` open, same pattern as `create()`'s append path). The user can still read every page that *did* finish indexing; forward-nav at the last cached page advances to the next spine instead of looping the same abort. `PageCache::clear()` also removes the marker so a future cache at the same path doesn't inherit an old crash signal. Diagnosed from MQuinteiroS's crash log — PC `0x4210eb83` was inside libgcc's `unwind-dw2.o`, the canonical signature of a thrown C++ exception with exceptions disabled.

**On-Device EPUB Indexer retired** — File browser → Right-press now goes straight to the delete-confirm dialog again (the 0.5.x behaviour). The "Index for faster reading" / "Delete" / "Cancel" popup added in 0.6.0 is no longer reachable from UI. The underlying code (`promptFileAction`, `FileActionMenuView`, `Screen::FileAction` handler, `startIndexing`, `indexOneSpineStep`, `finishIndexing`, `renderIndexDone`) is intentionally left in place but unreachable — a follow-up sweep can remove it. Rationale: cold-extend during indexing has the same `std::bad_alloc` failure mode as runtime cold-extend, and on the heap profile of a typical user's device, big books were a coin-flip. Pre-rendering on the desktop via `sumi.page/process` has none of these constraints (browser heap is effectively unlimited) and is the right path forward.

**C1 false-positive cleared** — `[GFX] BUG: clearScreen called from main task while cacheTask is still active. CONCURRENCY.md C1` was firing in serial logs after every normal page turn, even on user devices that hadn't actually violated the concurrency invariant. Root cause: `GfxRenderer::s_cacheTaskHandle_` was only cleared inside `stopBackgroundCaching()`, but that function's early-return-when-not-running path skipped the clear. When the bg cache task self-deleted at the end of its body (natural exit, not via `stop()`), the handle stayed pointing at a freed task. Next `clearScreen` from main → `xTaskGetCurrentTaskHandle() != s_cacheTaskHandle_` → spurious warning. Fix: clear the handle at the end of the bg task body itself, *and* always clear in `stopBackgroundCaching()` regardless of the `isRunning()` check. Cosmetic but it was masking the real signal in crash logs (was the bg task actually alive, or just a stale handle? now you can trust it).

**JPEGDEC MCU_SKIP wild-pointer crash (CrossPoint 1.3.0)** — `scripts/patch_jpegdec.py` runs as a pre-build script and idempotently patches `.pio/libdeps/*/JPEGDEC/src/jpeg.inl` at `JPEGDecodeMCU_P`. The pre-patch line computed `pMCU = &sMCUs[iMCU & 0xffffff]`; when `iMCU == MCU_SKIP` (-8), the bitmask produced index `0xFFFFF8` (16,777,208) and the resulting pointer landed ~33 MB past the 392-entry `sMCUs` array. If the progressive JPEG's first scan included AC coefficients (`iScanEnd > 0`), the AC decode loop wrote through this wild pointer and the device crashed with a store-access fault. Upstream commit 8628297 guarded the DC write (`pMCU[0]`) but not the AC writes at indices 1–63. The patch redirects `pMCU` to `sMCUs[0]` when `iMCU < 0` — writes to `sMCUs[1..63]` are harmless in MCU_SKIP because for `JPEG_SCALE_EIGHTH` only `sMCUs[0]` is read for output. Wired into both `[base]` and `[env:gh_release]` `extra_scripts`. Borrowed verbatim from CrossPoint Reader 1.3.0's `patch_jpegdec.py`.

**X3 grayscale washout fix (CrossPoint 1.3.0)** — On X3, every `HALF_REFRESH` now forces a full panel sync plus one settle pass via `EInkDisplay::requestResync(1)`. Without this, the X3's fast-diff path accumulates residue and grayscale AA pixels turn into haze after a handful of partial refreshes — the long-standing X3 caveat from the SUMI README ("Grayscale/AA rendering on X3 still uses the X4 path and may show artifacts until ported in a future release"). The driver-side plumbing (`_x3ForceFullSyncNext`, `_x3ForcedConditionPassesNext`, the consumer logic at `EInkDisplay.cpp:813,873,919`) was already present from earlier X3 bring-up work; nothing in `src/` had been calling `requestResync()`. Three call sites added: `GfxRenderer::displayBuffer()` (broad path, every refresh), `GfxRenderer::displayGrayBuffer()` (the 2-pass AA composite flush — the path that most reliably triggered washout), and `sumi::drivers::Display::flush()` (driver-level path that bypasses GfxRenderer). `requestResync()` self-gates on `_x3Mode`, so all three sites are no-ops on X4 — zero overhead for X4 users. X3 users can now leave Text Anti-Aliasing on.

**CJK no spurious hyphen at break (CrossPoint 1.3.0)** — `Hyphenator::appendSegmentPatternBreaks` used to push every break as `{offset, requiresInsertedHyphen=true}`. When a break landed between two CJK ideographs (a perfectly legal break point under UAX #14, no inserted glyph needed), the renderer would draw `-` between them — visible at wrap points in Chinese, Japanese, and Korean text. New `cjkAdjacentToBreak()` predicate (CJK Unified Ideographs, Extension A/B+, Compatibility Ideographs, Hiragana, Katakana, Hangul Syllables, Fullwidth forms) is checked on both sides of each break; if either side is CJK, `requiresInsertedHyphen=false`. Renderer side (`ParsedText.cpp:581`) already honored the flag — this was a one-end-of-the-pipe fix. The predicate is inlined into the hyphenation translation unit to avoid pulling `ScriptDetector` and `ParsedText`'s local copies into the build for one function (same pattern those files already use).

**Site updates that ship alongside 0.6.2.** `sumi.page/fonts` had two real bugs that this work surfaced — the on-screen guidance pointed users at `/fonts/` for `.bin` font installs (firmware reads `/config/fonts/`), and the `.epdfont` canvas-rendering math passed the wrong Y baseline to opentype.js, sending ascender glyphs to negative canvas Y where they got clipped. Both are fixed in the site repo. If you generated a custom font on the site between 0.6.0 and now and the device rendered "garbled" text — re-convert it after the site redeploys and the new file should render correctly.

## [0.6.1] — May 2026

Two-fix hotfix on top of 0.6.0.

**Books no longer reset to page 5 on every wake** — two coupled bugs in `PageCache`:
1. `RenderConfig::operator==` compared `showTables` but `PageCache::writeHeader` / `PageCache::load` never wrote or read it. The deserialized `fileConfig.showTables` always read back as its default (`true`), so any user with `settings.showTables = false` (Settings → Display → Show Tables off) hit a permanent config-mismatch on every cache load.
2. Fixing (1) by serializing `showTables` between `showImages` and `allowTallImages` revealed a second bug: `HEADER_SIZE` was calculated by summing the per-field sizes, and adding the new byte without updating that constant put `writeLut`'s `pageCount` write one byte too early — clobbering the high byte of `viewportHeight` on disk. On reload, `viewportHeight` read back wrong (low byte intact, high byte = pageCount-low-byte ≈ 10) and `Config mismatch` fired for every user, not just those with Show Tables off.

The visible effect of both was identical: `[CACHE] Config mismatch, invalidating cache` → section file deleted → fresh 5-page parse → reader clamped to the last cached page → user landed on "page 5" on every wake. Fix: serialize `showTables`; bump `HEADER_SIZE` 26 → 27; bump `CACHE_FILE_VERSION` 19 → 20 so existing caches do one clean rebuild on first wake post-upgrade. Reported by AbuMaia01 on 0.5.x; root cause caught with per-field diagnostic instrumentation that pinpointed the byte-offset corruption.

**Comic page washout fix** — 0.6.0's source-cleanup pass over-stripped `lib/EInkDisplay/src/EInkDisplay.cpp`, removing code paths that affected the grayscale render on real hardware. Visible symptom: comic pages rendered crisp for ~1 second then washed out to a grey haze (the v0.5.0 grayscale-washout bug re-introduced). EInkDisplay.cpp reverted to its 0.5.1 / 0.6.0-original state; comic rendering is back to smooth.

## [0.6.0-ramfix] — May 2026

The memory-rework release. The 0.5.1 firmware shipped a working reader for everyone with modest books, but heavy chapters + Bluetooth page-turners exposed a hard memory ceiling: the EPUB parser would either crash or get permanently stuck mid-book when NimBLE's ~48 KB resident allocation collided with chapter-parse heap demand. This release rebuilds the memory model from the bottom up — single-block linker-placed arena, glyph cache trimmed, dead allocation paths removed — then layers on surgical fixes to the reader's hot paths so BLE-active reading stays stable across long chapters. Ships a new on-device EPUB indexer that pre-builds every chapter's page cache for guaranteed instant loads (no parser at read-time), a multi-dictionary union lookup so installed dictionaries are searched in parallel, a format-aware in-reader settings overlay, and the largest audit pass to date (batches 1-10, ~70 hardening items landed).

**Memory Arena v2** — The 76 KB memory arena is now a single contiguous block placed in linker-managed `.bss` instead of three runtime-allocated chunks. The arena is permanent — never released, never re-allocated, never competes with heap fragmentation. Companion cuts that recovered the budget for BLE-active reading: 5.8 KB recovered by trimming `GfxRenderer::GlyphCache` from 64 to 16 entries; 8 KB recovered by removing the dead `Core::buf.decompress` scratch region; ~24 KB recovered overall from related dead allocation removals. Net effect: post-boot free heap rises from ~58 KB (0.5.1) to ~74 KB, and the arena's 76 KB never enters the heap pressure equation. The arena now also doubles as the storage for SumiBoy's ROM (`primaryBuffer` aliases the GB ROM buffer when SumiBoy is the active plugin) and the FreeRTOS task stack region for `cacheTask`, eliminating heap allocation for the background page-builder.

**On-Device EPUB Indexing** — File browser → right-press → "Index for faster reading" pre-builds every chapter's page cache on the device. After indexing, reads load instantly even with Bluetooth on — the parser never runs, the cold-extend path never fires, and the "stuck on page N" failure mode is structurally impossible because there's nothing to parse at read time. Releases the external font cache for the duration of the indexing run (frees 8-15 KB), iterates spines serially, renders "Chapter N of M" with progress bar before each spine's work, supports cancel via Back/Power mid-run, and persists the work to SD so it survives reboot. Tradeoff: font size / line spacing / hyphenation / alignment / show-images get baked in at index time — change any of them later and the affected caches invalidate and rebuild lazily on next read. The same code path is reachable from `sumi.page/process` via BLE transfer; the on-device path is for users without a paired desktop.

**Format-Aware In-Reader Settings** — `InReaderSettingsView::SettingDef` gained a `validForXtc` flag. On XTC books (whether pre-rendered with `sumi.page/process` or indexed on-device), the in-reader settings overlay now hides the controls that have no effect on a pre-laid-out format: Font, Font Size, Text Layout, Line Spacing, Alignment, Hyphenation, Show Images. The remaining controls — Look up Word, Lookup History, Bookmarks, Anti-Aliasing, Status Bar, Text Darkness, Bluetooth — stay visible because they still affect runtime rendering. Navigation skips hidden rows via a `slot/visibleIndex` mapping in `moveUp/moveDown/ensureVisible`, so Up/Down feels exactly the same as a flat list.

**Multi-Dictionary Union Search** — Drop any number of StarDict dictionaries under `/dictionary/<name>/` and every lookup now searches all of them in parallel, returning labelled definitions from each match. Previously the device required choosing one "active" dictionary and would only consult it; switching dicts mid-reading was a five-step settings dive. New `Dictionary::lookupAll(word)` and `Dictionary::findSimilarAll(word, max)` walk the available-dict list, calling `setActive()` per dict so only one sparse offset cache (~30-80 KB) is resident at a time. Results are rendered as labelled sections in the existing `DictDefinitionView` (`Section { sourceLabel, body }`), separated by horizontal rules with the dictionary's display name above each entry. Stem fallback runs in a second pass after direct misses across every dict — one stem hit per dict, max. The standalone Dictionary plugin and the in-reader long-press-word-lookup both call `lookupAll`, so the union behaviour is consistent everywhere. Use case: GCIDE for general English, Etymology Online for word history, a medical dictionary for technical terms — all three searched per lookup, no "which one is active right now" cognitive load.

**Power Button → In-Reader Settings** — Long-press Center (the OK side of the Back/Select rocker) was supposed to open the in-reader settings overlay, but the hardware rocker can't sustain a clean ≥700 ms press reliably — small ADC drift drops the press below the detection threshold, resetting the timer, and the eventual release fires as a short-press (which opens the TOC). With `shortPwrBtn == PowerIgnore` (the default), short-press Power in the reader now opens the in-reader settings overlay. Existing customizations (PageTurn, Refresh) still work and override the new default. Long-press Center remains wired up for users whose hardware can hit the threshold cleanly.

**Render brackets the bg cache task** — `ReaderState::render()` now calls `stopBackgroundCaching()` at the top and `startBackgroundCaching(core)` at the bottom, so the main task never touches the `GfxRenderer` while the bg cache task is alive. `ReaderState::applyNavResult()` no longer starts the bg cache task prematurely when a render is coming — it defers to render's tail. Fixes the CONCURRENCY.md C1 violations that were rate-limit-spammed into the serial log, AND meaningfully reduces peak heap during cache extension under BLE pressure (bg task and main render were previously both holding parser/render state at the same time, doubling peak usage). Verified empirically with the new `[HP] nav` / `[HP] render` heap-timeline instrumentation: with BLE active, the same chapter that previously capped at 15 cached pages now reaches 20-30 pages before the parser's safety threshold trips.

**Font Release During Cold Extend** — When `createOrExtendCache` runs (whether from the on-device indexer or the runtime cold-extend path), the external `.bin` font's glyph cache is temporarily unloaded for the duration of the parse and reloaded immediately after. The parser doesn't render glyphs — it only measures widths via `getTextWidth`, which falls back gracefully to built-in font metrics during the parse. Recovers 8-15 KB of heap during the parse window, enough to push the parser's safety threshold further along on a fragmented heap. The next render after the parse sees the font reloaded, so CJK glyphs render correctly.

**Session-Complete on Failed Extend** — When a cold-extend produces zero new pages (parser bailed at the `< 4096 B` safety threshold in `ChapterHtmlSlimParser::startElement`), `PageCache::extend` now marks the cache `isPartial_ = false` in memory only — the on-disk header still says partial. Effect: navigation treats the chapter as complete for this session, so pressing Forward at the last cached page advances to the next spine instead of looping on a doomed re-extend. Next reboot reloads the disk header (still partial) and gets a fresh chance to extend under whatever heap conditions exist then. Fixes the "press Forward, screen shows the same page, press again, still the same page, BLE feels broken" UX failure mode without permanently truncating any chapter.

**[HP] Heap Timeline Instrumentation** — `ReaderState::applyNavResult` and `ReaderState::render` emit `[HP] nav:` / `[HP] render:` lines on every page turn with `spine`, `page`, `flat` (page index), `free` (total free heap), `largest` (largest contiguous block), and `ble` (BLE-connected flag). Used during this release's stability work to build a per-turn timeline distinguishing fragmentation (largest collapses, free stable) from leaks (free trends down). Stays in the build because the cost is one Serial.printf per turn and the diagnostic value is high — a stuck reader's serial log immediately shows whether it's a parser bail vs a state machine wedge.

**File Browser Action Popup** — Right-press on a file in the browser now opens a 3-option popup: "Index for faster reading", "Delete", "Cancel". Replaces the previous straight-to-confirm-delete flow. The Index entry is the new on-device indexer (see above); Delete routes into the same confirmation dialog as before; Cancel returns to browse. The popup defaults to Index so the new action is immediately discoverable; users who hit Right intending to delete see the menu, read the new option, and can press Down to reach Delete.

**Long EPUB Paragraph Truncation Fix** — `ChapterHtmlSlimParser` previously truncated paragraphs that spanned more pages than the `maxPages` soft-stop budget. The XML parser would suspend mid-text-block on `completePageFn returning false`, but by the time `XML_StopParser` actually suspended, control had unwound to `startNewTextBlock` which `reset()` the in-progress text block — losing any words extractLine had already drained. The fix turns the budget-hit signal into a *soft-stop*: `addLineToPage` sets `pendingMaxPagesSuspend_` instead of suspending immediately; `layoutAndExtractLines` runs to completion and fully drains the current block; `makePages` suspends XML at the clean end-of-block boundary where `currentTextBlock` is empty and `startNewTextBlock` can safely reset it. EOF cleanup unconditionally emits the trailing partial page so the soft-stop doesn't truncate the last paragraph either. Reported by ciceronianus and AbuMaia01 — "certain paragraphs end abruptly... changing font size moves which paragraphs are cut." New host test (`SoftSuspendParagraphTest.cpp`) asserts every input word survives across multi-batch parses with budget-1, covering single-block, multi-block, and 50-line-paragraph scenarios.

**Audit Batches 1-10 (~70 items landed)** — The largest hardening pass in SUMI's history. Highlights:
- **Atomic writes everywhere**: settings, recent books, global bookmarks, reading stats, todo list, book overrides, Lua `sumi.writeFile()`, plugin saves, GB save state, GB cheats. All `/.sumi/` and `/games/` writes now use the 3-rename rotation protocol (write-tmp → rename-canonical-to-bak → rename-tmp-to-canonical → delete-bak) per `ATOMIC_WRITE_DESIGN.md`. Crash mid-write leaves at least one valid copy. `SDCardManager::recoverAtomicWrites()` extended from `/.sumi` to also scan `/notes`, `/data`, `/custom`, `/games`.
- **nothrow new at every top-level allocation site**: parser, page cache, plugin factories (24 sites), graphics/font allocations, ReaderState parser/cache, Epub/Xtc top-level. With `-fexceptions` off, raw `new` aborts on OOM; nothrow returns nullptr and lets the caller bail cleanly.
- **CRC32 trailer on 8 `.bin` formats**: settings.bin, recent.bin, library.bin, reading_stats.bin, achievements.bin, global_bookmarks.bin, overrides.bin, reader_guard.bin. Corrupt files no longer crash the device on load — bad CRC means we reject the file and rebuild from defaults.
- **Bounds defense on every disk-read count field**: `FileListState` dir scan capped during collection (not after); `Dictionary::readWord` bounded against `kMaxWordBytes`; `HomeState` recent-book selection has both upper and lower bounds; `ChapterListView::setCurrentChapter` clamps input; `DictWordSelectView::render` and `findClosestWord` bounds-defended; Library index untrusted count cap.
- **`InputDrainGuard`** at the top of `ReaderState::enter()` and `HomeState::enter()` drains queued button events and waits up to 500 ms for all physical buttons to release. Fixes the long-standing issue where holding Power to wake from sleep would leak that press into the new state as an accidental page turn.
- **Lua sandbox escape closed**: `io`, `os`, `debug`, `package` libraries are now stripped from the Lua VM so a malicious plugin can't reach the SD card outside its sandbox or introspect the C++ stack.
- **NimBLE callback fan-out**: `BleHostCallbacks` dispatcher replaces the fragile "later `setCallbacks()` caller wins" arrangement between BleFileTransfer and BleBridge. Adding a new BLE subsystem in the future no longer silently breaks an existing one.
- **`merge_firmware.py` SHA256 sidecar**: every release bin now has a `.sha256` next to it so the web flasher can verify the download before writing — closes the network-corruption-bricks-device failure mode.
- **Crash log appends** instead of overwriting: `crash.log` now keeps history of crashes across boots instead of losing prior context on the next abort.
- **`build_html.py`** random per-run placeholder so deterministic build hash drift is caught (rather than masked).
- **3-rename rotation for GB savestate + cheats**: same atomic-write semantics SUMI's core state files use.

**Dual-Bin Release Packaging** — The flashable `sumi-v0.6.0-ramfix-full.bin` is now a true dual-app merge: bootloader at 0x0, partition table at 0x8000, SUMI app at 0x10000 (~3 MB), SumiBoy app at 0x650000 (~1.65 MB). Single file, ~8.3 MB, flashes via web flasher or `esptool write_flash 0x0`. Previously the SUMI bin only contained app0 and SumiBoy had to be flashed separately as a second step at offset 0x650000, which was easy to forget and led to "the SumiBoy menu item does nothing" support tickets. The merge step lives in `scripts/merge_firmware.py` post-action when the SumiBoy raw payload is present at `firmware/sumiboy-v1.1.0.bin` (staged from the X4_emulator project).

**EPUB / Reader Stability Fixes** — Eleven defensive fixes in EPUB and reader paths:
- `parseTocNcxFile` and `parseTocNavFile` `FsFile::read` signed-return bugs fixed (the last two surviving from the overnight sweep).
- "Paragraph words silently dropped during emergency split" — text emitted in the wrong order if an emergency split fired mid-paragraph.
- "Don't clear crash guard on first-page render" — cover render is not a sufficient health signal; the background page cache task can still crash after cover. Crash guard is now cleared only on clean exit or first user-initiated navigation.
- "Skip EPUB metadata pre-load when heap is fragmented" — defensive bailout in HomeState if metadata pre-load can't be done safely.
- `Hyphenator` language reset when switching from EPUB to TXT/Markdown.
- ZipFile + EPUB chapter HTML parser short-read detection.
- Streaming manifest lookup from temp file when the manifest is too big to hold in heap.
- Delete corrupt cached BMPs so retries re-extract instead of looping on the same corrupt file.

**CJK / UTF-8 Safety Improvements**:
- `.bin` fonts (CJK fallbacks) now selectable from the in-reader Font picker; previously the picker filtered them out and CJK rendered as `?`.
- CJK-safe ROM name extraction in SumiBoy save paths.
- CJK-safe title truncation in the status bar.
- CJK-safe quote truncation in the Daily Quote plugin.
- CJK path safety in various filesystem-touching paths.

**X3 Hardware Fixes**:
- `main.cpp` screenshot BMP no longer uses hardcoded X4 dimensions — works correctly on X3's 792×528 panel.
- `ChessGame` + `Sudoku` now honor passed-in screen dimensions instead of assuming X4 800×480.

**Plugins**:
- `PluginListState` category overflow fix (more than `MAX_PLUGINS_PER_CATEGORY` plugins in one category was truncating).
- `SumiBoyEmulator` ROM read error handling — corrupt or short ROM file now bails cleanly instead of aborting.
- SumiBoy ROM backed by arena primary buffer (32 KB) — no more separate heap allocation for the ROM, eliminates the OOM path that broke "open Tetris" on fragmented heap.
- SumiBoy menu flattened from category-grouped to flat list (CrossPoint pattern); theme-cache defrag fix.
- "Boot crash resilience" — `readerLoadAttempts` in `/.sumi/reader_guard.bin` skips reader auto-resume after 3 attempts.

## [0.5.1] — April 2026

**Plugin Bridge** — Lua plugins can now exchange live JSON messages with a Web-Bluetooth client on sumi.page via a new BLE service (UUID `19B10100-E8F2-537E-4F6C-D104768A1214`, TX notify + RX write characteristics). New `bridge` global table in plugin scripts exposes `bridge.publish(topic, data)`, `bridge.on(topic, fn)`, `bridge.connected()`, `bridge.keep_awake(seconds)` — plugins can push events to a laptop (doorbell, notifications, step counters), receive text imports or remote commands, and hold the device awake while paired. `keep_awake` hooks into main.cpp's auto-sleep gate (max 3600s per call, renewable from `update()`, auto-clears on plugin exit). `LuaBridgeBindings.h` converts Lua values ↔ ArduinoJson variants so handlers receive properly decoded tables. Bounded 4-slot ring buffers in each direction keep BLE callbacks non-blocking. New sumi.page/bridge page has a **Build** tab (prompt generator that inlines the full Lua + Bridge API surface — paste into any AI, get back runnable plugins) and a **Run** tab (connect, raw JSON console, Text Importer helper, Doorbell receiver with WebAudio chime + desktop notifications, user-paste Custom Handler slot running in a sandbox="allow-scripts" iframe with a `bridge.publish`/`bridge.on` postMessage shim). Canonical `doorbell.lua` + `doorbell.html` example pair ships under `docs/plugin_examples/`. Protocol reference at `docs/PLUGIN_BRIDGE.md`; self-contained AI authoring prompt at `docs/PLUGIN_AUTHORING_PROMPT.md`. Unlocks the "any user builds whatever plugin they want, deploys it themselves, no feature requests needed" workflow.

**Xteink X3 Hardware Support** — One firmware, two devices. SUMI now auto-detects whether it's running on an Xteink X4 (800×480, SSD1677 panel) or the smaller Xteink X3 (792×528, different display controller with BQ27220 fuel gauge, DS3231 RTC, QMI8658 IMU). Detection runs I2C fingerprint probes at first boot, caches the result in NVS flash at `cphw:dev_det`, and subsequent boots read the cache without probing. An NVS override key (`cphw:dev_ovr`: 0=auto, 1=force X4, 2=force X3) handles recovery. New `lib/HardwareDetect` module owns the probe logic. `EInkDisplay` refactored from compile-time `constexpr DISPLAY_WIDTH = 800` to runtime `displayWidth` + `setDisplayX3()` — the framebuffer is sized for the larger of X4/X3 (MAX_BUFFER_SIZE=52272), `getScreenWidth/Height` return runtime values, all UI auto-adapts. Full X3 refresh path ported from Crosspoint master (img LUTs for full sync, full LUTs for fast diff, 0x13 write + 0x10 RED RAM sync + 0x04 power-on + 0x12 trigger + post-condition passes). X3 busy-wait polarity (LOW-while-busy) differs from X4 (HIGH-while-busy) — `waitWhileBusy()` dispatches on `_x3Mode`. Ported from the Crosspoint master X3 merge; grayscale/AA path on X3 still uses X4 commands and may show artifacts until tested on real X3 hardware (workaround: disable Anti-Aliasing in Reader settings).

**I18n Framework** — `lib/I18n` with 13-language enum (EN, ES, FR, DE, PT, IT, RU, PL, NL, JA, ZH, KO, AR) and 45 translatable string IDs covering home, settings, reader, file browser, plugins, sleep, system info. English is complete, Spanish/French/German/Portuguese/Japanese have partial coverage, fallback to English for nullptr entries. `_tr()` macro (named with underscore to avoid conflict with ChessGame's Move::tr member). Language selector added to Settings → Device. Initial pass converted ~40 of 222 identified UI strings; remaining strings can migrate incrementally without blocking other work.

**Reading Statistics with Monthly Heatmap** — `lib/ReadingStats` tracks per-book reading time, session counts, pages turned, books started/finished, and daily reading minutes in a 90-entry circular buffer. Reading Stats plugin (`src/plugins/ReadingStatsApp.h`) has two screens: Overview (total time, sessions, pages, streaks, books) and Heatmap (monthly calendar with 4-level heat cells showing 0 / <10m / 10-30m / 30-60m / >60m, month navigation, "Less [...] More" legend). **ErrorState recovery** — ErrorState previously tried to transition to FileList, which doesn't exist in READER boot mode. Now restarts into full UI mode instead of getting stuck.

**Flashcard decks visible in file browser** — `.tsv` and `.csv` files now appear in the file browser. Selecting one launches the Flashcards app directly instead of trying to open it in the reader (which would fail and trigger a restart loop through ErrorState).

**Web flasher fixes** — corrected SumiBoy manifest offset to 0x650000 (6619136 decimal). Fixed partition diagram (was showing 3MB partitions at 0x310000, actual layout is 6.25MB at 0x650000). Removed non-functional erase checkbox, added clear guidance to use "Erase device" for SUMI installs and skip it for SumiBoy. Fixed stale version/size tags.

## [0.3.2] — 2026-02-15

Table rendering and content hint pipeline. Tables now render as actual column-aligned layouts with borders instead of `[Table omitted]` — the parser collects cell data, calculates proportional column widths, and positions cell text with `+--+` grid borders. Header rows render in bold with a separator line. Cells truncate gracefully when the table is wider than the screen.

Also added content hints: EPUBs from sumi.page now embed `dc:subject` tags that tell the firmware what kind of content it's looking at. The OPF parser picks them up, stores them in the book cache (v6) and library index (v2), and the file browser renders labels like `MNGA 42%` or `NEWS` next to your reading progress. The converter, newspaper builder, and web clipper all tag their output automatically. EPUB passthrough injects `sumi:book` if nothing's there.
